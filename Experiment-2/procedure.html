Experiment: Design of Entity–Relationship (ER) Diagram and Relational Mapping

SRM Institute of Science and Technology
Department of Computer Science
Procedure

Step-by-Step Procedure

Follow these steps to design an ER diagram and map it to a relational schema.

  1. Identify the Problem Domain
    Choose a real-world scenario (e.g., University Management System, Library Management System, Hospital Management System). Clearly define the scope and requirements of the system.
  2. Identify Entities
    List all entities in the system. Entities are real-world objects or concepts that can be distinctly identified. Examples: Student, Course, Department, Employee, Book, etc.
    • Write down all entities
    • Ensure each entity is distinct and meaningful
  3. Identify Attributes for Each Entity
    For each entity, list all attributes that describe it. Classify attributes as:
    • Simple attributes (atomic values)
    • Composite attributes (can be divided into sub-attributes)
    • Single-valued or multi-valued attributes
    • Derived attributes (calculated from other attributes)
    Example: Student entity with attributes: student_id (primary key), name, email, date_of_birth, address (composite).
  4. Identify Primary Keys
    For each entity, identify the primary key attribute(s) that uniquely identify each instance. Primary keys are underlined in ER diagrams. 
    Example:
Student: student_id (primary key)
Course: course_id (primary key)
  5. Identify Relationships
    Identify relationships between entities. Determine the type of relationship:
    • One-to-One (1:1): Each instance of Entity A relates to at most one instance of Entity B
    • One-to-Many (1:N): One instance of Entity A relates to many instances of Entity B
    • Many-to-Many (M:N): Many instances of Entity A relate to many instances of Entity B
    Example:
Student ENROLLS Course (M:N)
Department HAS Employee (1:N)
  6. Determine Cardinality and Participation
    For each relationship, specify:
    • Cardinality: The number of instances (1, N, M)
    • Participation: Total (double line) or Partial (single line)
    Example:
Every employee MUST belong to a department (total participation)
Not every student enrolls in a course (partial participation)
  7. Identify Weak Entities (if any)
    Identify entities that cannot be uniquely identified without a relationship to another entity. Weak entities are represented by double rectangles and require an identifying relationship. 
    Example:
Dependant (weak entity) depends on Employee (strong entity)
  8. Draw the ER Diagram
    Using standard ER notation, draw the complete ER diagram:
    • Rectangles for entities
    • Ovals for attributes
    • Diamonds for relationships
    • Double rectangles for weak entities
    • Underline primary keys
    • Indicate cardinality on relationship lines
    • Use double lines for total participation
  9. Map Strong Entities to Relations
    For each strong entity, create a relation (table):
    • Entity name becomes table name
    • All attributes become columns
    • Primary key becomes the table's primary key
    • Composite attributes are flattened into separate columns
    Example:
Entity: Student(student_id, name, email, DOB)
Relation: Student(student_id, name, email, date_of_birth)
          Primary Key: student_id
  10. Map Weak Entities to Relations
    For each weak entity, create a relation including:
    • All attributes of the weak entity
    • Primary key of the owner entity as a foreign key
    • Combined primary key (owner's key + weak entity's partial key)
    Example:
Weak Entity: Dependant(dep_name, age, relationship)
Owner Entity: Employee(emp_id, name)
Relation: Dependant(emp_id, dep_name, age, relationship)
          Primary Key: (emp_id, dep_name)
          Foreign Key: emp_id references Employee
  11. Map 1:1 Relationships
    For one-to-one relationships, include the foreign key in either of the participating entities: 
    Example:
Person(ssn, name) OWNS Passport(passport_no, country)
Option 1: Person(ssn, name, passport_no)
Option 2: Passport(passport_no, country, ssn)
  12. Map 1:N Relationships
    For one-to-many relationships, include the foreign key in the entity on the "many" side: 
    Example:
Department(dept_id, dept_name) HAS Employee(emp_id, name, salary)
Result: Employee(emp_id, name, salary, dept_id)
        Foreign Key: dept_id references Department
  13. Map M:N Relationships
    For many-to-many relationships, create a separate relation with foreign keys from both entities: 
    Example:
Student(student_id, name) ENROLLS Course(course_id, title)
Result: Enrollment(student_id, course_id, grade, semester)
        Primary Key: (student_id, course_id)
        Foreign Keys: student_id, course_id
  14. Handle Multi-valued Attributes
    For multi-valued attributes, create a separate relation: 
    Example:
Employee(emp_id, name) with multi-valued attribute: phone_numbers
Result: Employee(emp_id, name)
        Employee_Phone(emp_id, phone_number)
        Foreign Key: emp_id references Employee
  15. Verify the Relational Schema
    Review the mapped relational schema to ensure:
    • All entities are mapped to relations
    • All relationships are properly represented
    • Primary keys and foreign keys are correctly identified
    • No information is lost in the mapping process

Expected Outcomes

  • Ability to identify entities, attributes, and relationships from a problem domain
  • Understanding of ER diagram notation and symbols
  • Knowledge of cardinality and participation constraints
  • Skill in mapping ER diagrams to normalized relational schemas
  • Understanding of how to handle weak entities, composite attributes, and multi-valued attributes
