ER Diagram Design Simulation

Follow the steps below to understand how to design an ER diagram and map it to a relational schema. This simulation uses a University Management System as an example.

Step 1: Entities and Attributes

In this step, we identify the entities and their attributes. Entities are represented as rectangles, and attributes are listed within each entity. Primary keys are underlined.

Key Points:

  • Each entity represents a real-world object (Student, Course, Department)
  • Attributes describe the properties of each entity
  • Primary keys (underlined) uniquely identify each entity instance
  • All attributes shown here are simple attributes (atomic values)

Relationship Types Explained:

  • ENROLLS (M:N): Many students can enroll in many courses. This is a many-to-many relationship.
  • OFFERS (1:N): One department can offer many courses, but each course belongs to one department.
  • HAS (1:N): One department has many students, but each student belongs to one department.

Step 3: Relational Mapping

This step shows how the ER diagram is mapped to relational database tables. Each entity becomes a table, and relationships are represented using foreign keys or separate tables.

Enrollment Table (from M:N Relationship: Student ENROLLS Course)

student_id
FK, PK		 course_id
FK, PK		 grade semester
S001 C001 A Fall 2024
S001 C002 B Fall 2024
S002 C001 A Spring 2025

Mapping Rule: M:N relationships → Separate table with foreign keys from both entities. Composite primary key: (student_id, course_id).

Mapping Summary:

  • Strong Entities: Each entity becomes a table with all its attributes as columns.
  • 1:N Relationships: Foreign key is placed in the table on the "many" side (Student.dept_id, Course.dept_id).
  • M:N Relationships: Create a separate table (Enrollment) with foreign keys from both entities.
  • Primary Keys: Underlined attributes in ER diagram become primary keys in tables.
  • Foreign Keys: Represent relationships and maintain referential integrity.
