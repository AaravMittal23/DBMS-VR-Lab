Database Normalization Simulation

Follow the steps below to understand how to normalize a database relation using functional dependencies. This simulation demonstrates the normalization process from an unnormalized relation through 1NF, 2NF, and 3NF.

Original Unnormalized Relation

We start with an unnormalized relation that contains redundancy and potential anomalies. This relation represents student course enrollment information.

Student_Course Relation (Unnormalized)

student_id student_name dept_id dept_name course_id course_name instructor grade
S001 John Doe D01 Computer Science C001 Database Systems Dr. Smith A
S001 John Doe D01 Computer Science C002 Data Structures Dr. Johnson B
S002 Jane Smith D01 Computer Science C001 Database Systems Dr. Smith A
S003 Bob Wilson D02 Mathematics C003 Calculus Dr. Brown B

Functional Dependencies Identified:

student_id → student_name, dept_id
dept_id → dept_name
course_id → course_name, instructor
student_id, course_id → grade

Problems Identified:

  • Data Redundancy: Student name and department information is repeated for each course enrollment
  • Update Anomaly: Changing a student's name requires updating multiple rows
  • Insertion Anomaly: Cannot add a new department without a student
  • Deletion Anomaly: Deleting a student's last enrollment may lose department information
  • Partial Dependency: student_name depends only on student_id (part of composite key)
  • Transitive Dependency: dept_name depends on dept_id, which depends on student_id

Step 1: First Normal Form (1NF)

A relation is in 1NF if all attributes contain atomic (indivisible) values. In our example, all values are already atomic, so the relation is already in 1NF.

Student_Course Relation (1NF)

Primary Key: (student_id, course_id)

student_id
PK		 student_name dept_id dept_name course_id
PK		 course_name instructor grade
S001 John Doe D01 Computer Science C001 Database Systems Dr. Smith A
S001 John Doe D01 Computer Science C002 Data Structures Dr. Johnson B
S002 Jane Smith D01 Computer Science C001 Database Systems Dr. Smith A
S003 Bob Wilson D02 Mathematics C003 Calculus Dr. Brown B

1NF Status:

✓ All attributes contain atomic values
✓ No repeating groups
✓ Relation is in 1NF

Note: The relation still has redundancy and anomalies. We need to proceed to 2NF.

Step 2: Second Normal Form (2NF)

A relation is in 2NF if it is in 1NF and every non-key attribute is fully functionally dependent on the primary key. We need to eliminate partial dependencies. Here, student_name and dept_id depend only on student_id (partial dependency).

Enrollment Relation (2NF)

Primary Key: (student_id, course_id)

student_id
PK, FK		 course_id
PK, FK		 grade
S001 C001 A
S001 C002 B
S002 C001 A
S003 C003 B

Student Relation (2NF)

Primary Key: student_id

student_id
PK		 student_name dept_id
FK		 dept_name
S001 John Doe D01 Computer Science
S002 Jane Smith D01 Computer Science
S003 Bob Wilson D02 Mathematics

⚠️ Still Not Fully Normalized:

The Student relation still has a transitive dependency: student_id → dept_id → dept_name
This violates 3NF. We need to decompose further.

Step 3: Third Normal Form (3NF)

A relation is in 3NF if it is in 2NF and no non-key attribute is transitively dependent on the primary key. We eliminate the transitive dependency by separating Department information.

Enrollment Relation (3NF)

Primary Key: (student_id, course_id)

student_id
PK, FK		 course_id
PK, FK		 grade
S001 C001 A
S001 C002 B
S002 C001 A
S003 C003 B

Student Relation (3NF)

Primary Key: student_id

student_id
PK		 student_name dept_id
FK
S001 John Doe D01
S002 Jane Smith D01
S003 Bob Wilson D02

Department Relation (3NF)

Primary Key: dept_id

dept_id
PK		 dept_name
D01 Computer Science
D02 Mathematics

Course Relation (3NF)

Primary Key: course_id

course_id
PK		 course_name instructor
C001 Database Systems Dr. Smith
C002 Data Structures Dr. Johnson
C003 Calculus Dr. Brown

✓ Normalization Complete (3NF)

  • ✓ All relations are in 3NF
  • ✓ No partial dependencies
  • ✓ No transitive dependencies
  • ✓ Data redundancy eliminated
  • ✓ Anomalies removed
  • ✓ Lossless decomposition achieved
  • ✓ Functional dependencies preserved

Result: The database is now properly normalized with four relations that maintain data integrity and eliminate redundancy while preserving all information from the original relation.
