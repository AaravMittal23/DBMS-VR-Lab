Experiment: Database Normalization Using Functional Dependencies

SRM Institute of Science and Technology
Department of Computer Science
Procedure

Step-by-Step Procedure

Follow these steps to perform the experiment on Database Normalization Using Functional Dependencies.

  1. Start with an Unnormalized Table
    Create or analyze an unnormalized table. Example: Student_Course(student_id, name, course_id, course_name, grade, instructor, dept_name)
    • Observe the structure and identify potential problems
    • Note any repeating groups or multivalued attributes
    • Identify data redundancy
  2. Identify Functional Dependencies
    Analyze the table and list all functional dependencies: 
    Example:
student_id → name
course_id → course_name, instructor
student_id, course_id → grade
course_id → dept_name
    Document all FDs clearly, noting which are partial or transitive.
  3. Identify Keys
    Determine candidate keys, primary key, and superkeys:
    • Find all superkeys (sets of attributes that uniquely identify tuples)
    • Identify candidate keys (minimal superkeys)
    • Choose a primary key
    Example:
Superkeys: (student_id, course_id), (student_id, course_id, grade), etc.
Candidate Key: (student_id, course_id)
Primary Key: (student_id, course_id)
  4. Check for 1NF Violations
    Verify if the table is in First Normal Form:
    • Check for multivalued attributes or repeating groups
    • Ensure all attributes contain atomic values
    • If not in 1NF, decompose to eliminate repeating groups
    If not in 1NF:
- Separate repeating groups into new tables
- Ensure each cell contains a single value
  5. Normalize to 1NF
    If the table violates 1NF, decompose it: 
    Example:
Before 1NF: Student(student_id, name, courses[])
After 1NF: Student(student_id, name)
           Enrollment(student_id, course_id)
    Verify that all attributes are atomic and there are no repeating groups.
  6. Check for 2NF Violations
    Verify if the table is in Second Normal Form:
    • Check if the table is in 1NF
    • Identify partial dependencies (non-key attributes depending on part of composite key)
    • If partial dependencies exist, the table is not in 2NF
    Example - Not in 2NF:
Order(order_id, product_id, product_name, quantity)
FD: order_id, product_id → quantity
    product_id → product_name (partial dependency)
  7. Normalize to 2NF
    If the table violates 2NF, decompose it: 
    Example:
Before 2NF: Order(order_id, product_id, product_name, quantity)
After 2NF: Order(order_id, product_id, quantity)
           Product(product_id, product_name)
    Ensure all non-key attributes are fully functionally dependent on the primary key.
  8. Check for 3NF Violations
    Verify if the table is in Third Normal Form:
    • Check if the table is in 2NF
    • Identify transitive dependencies (non-key attribute depends on another non-key attribute)
    • If transitive dependencies exist, the table is not in 3NF
    Example - Not in 3NF:
Student(student_id, name, dept_id, dept_name, location)
FD: student_id → dept_id → dept_name, location
    (transitive dependency)
  9. Normalize to 3NF
    If the table violates 3NF, decompose it: 
    Example:
Before 3NF: Student(student_id, name, dept_id, dept_name, location)
After 3NF: Student(student_id, name, dept_id)
           Department(dept_id, dept_name, location)
    Ensure no non-key attribute is transitively dependent on the primary key.
  10. Check for BCNF Violations
    Verify if the table is in Boyce-Codd Normal Form:
    • Check if the table is in 3NF
    • Verify that every determinant is a candidate key
    • If a non-key attribute determines another attribute, it violates BCNF
    Example - Not in BCNF:
Course(course_id, instructor, time, room)
FD: course_id → instructor
    instructor, time → room
    (instructor is not a key)
  11. Normalize to BCNF (if needed)
    If the table violates BCNF, decompose it: 
    Example:
Before BCNF: Course(course_id, instructor, time, room)
After BCNF: Course(course_id, instructor, time)
            Room(instructor, time, room)
    Ensure every determinant is a candidate key.
  12. Verify Lossless Decomposition
    Ensure that the decomposition is lossless (no information is lost):
    • Check that joining the decomposed tables recovers the original table
    • Verify that all functional dependencies are preserved
    • Ensure no spurious tuples are generated
  13. Create Normalized Tables
    Create the normalized tables in your database: 
    CREATE TABLE Student (
    student_id VARCHAR(10) PRIMARY KEY,
    name VARCHAR(50),
    dept_id VARCHAR(10),
    FOREIGN KEY (dept_id) REFERENCES Department(dept_id)
);

CREATE TABLE Course (
    course_id VARCHAR(10) PRIMARY KEY,
    course_name VARCHAR(50),
    instructor VARCHAR(50),
    dept_id VARCHAR(10),
    FOREIGN KEY (dept_id) REFERENCES Department(dept_id)
);

CREATE TABLE Enrollment (
    student_id VARCHAR(10),
    course_id VARCHAR(10),
    grade CHAR(2),
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES Student(student_id),
    FOREIGN KEY (course_id) REFERENCES Course(course_id)
);

CREATE TABLE Department (
    dept_id VARCHAR(10) PRIMARY KEY,
    dept_name VARCHAR(50),
    location VARCHAR(50)
);
  14. Insert Sample Data
    Insert sample data into the normalized tables and verify:
    • Data integrity is maintained
    • No redundancy exists
    • All relationships are properly represented
  15. Compare Before and After
    Compare the unnormalized and normalized versions:
    • Note the reduction in redundancy
    • Verify that anomalies are eliminated
    • Check that all information is preserved
    • Understand the trade-offs (more tables vs. less redundancy)

Expected Outcomes

  • Understanding of functional dependencies and their identification
  • Knowledge of normalization forms (1NF, 2NF, 3NF, BCNF)
  • Ability to identify and eliminate partial and transitive dependencies
  • Skill in decomposing unnormalized tables into normalized schemas
  • Understanding of lossless decomposition and dependency preservation
  • Recognition of the benefits and trade-offs of normalization
