Introduction to Database Normalization

Database normalization is a systematic approach to organizing data in a database to eliminate redundancy and reduce anomalies. It involves decomposing a relation into smaller, well-structured relations based on functional dependencies. Normalization ensures data integrity, reduces storage space, and prevents update, insertion, and deletion anomalies.

Functional Dependencies

A functional dependency (FD) is a constraint between two sets of attributes in a relation. If attribute set X functionally determines attribute set Y (denoted as X → Y), then for any two tuples with the same values in X, they must have the same values in Y. 

Example:
If student_id → name, email
Then: If two tuples have the same student_id, 
      they must have the same name and email.

Key Terms:

  • Determinant: The left side of a functional dependency (X in X → Y)
  • Dependent: The right side of a functional dependency (Y in X → Y)
  • Trivial FD: Y is a subset of X (e.g., AB → A)
  • Non-trivial FD: Y is not a subset of X
  • Full FD: Y is functionally dependent on X but not on any proper subset of X
  • Partial FD: Y is functionally dependent on a proper subset of X

First Normal Form (1NF)

A relation is in 1NF if every attribute contains only atomic (indivisible) values. There should be no repeating groups or multivalued attributes. Each cell must contain a single value. 

Example - Not in 1NF:
Student(student_id, name, courses)
courses: "DBMS, OS, Networks"

In 1NF:
Student(student_id, name)
Enrollment(student_id, course)

Second Normal Form (2NF)

A relation is in 2NF if it is in 1NF and every non-key attribute is fully functionally dependent on the primary key. No partial dependencies are allowed. 

Example - Not in 2NF:
Order(order_id, product_id, product_name, quantity)
FD: order_id, product_id → quantity
    product_id → product_name (partial dependency)

In 2NF:
Order(order_id, product_id, quantity)
Product(product_id, product_name)

Third Normal Form (3NF)

A relation is in 3NF if it is in 2NF and no non-key attribute is transitively dependent on the primary key. All non-key attributes must depend directly on the primary key. 

Example - Not in 3NF:
Student(student_id, name, dept_id, dept_name, location)
FD: student_id → dept_id → dept_name, location
    (transitive dependency)

In 3NF:
Student(student_id, name, dept_id)
Department(dept_id, dept_name, location)

Boyce-Codd Normal Form (BCNF)

A relation is in BCNF if it is in 3NF and every determinant is a candidate key. BCNF is stricter than 3NF and eliminates all redundancy based on functional dependencies. 

Example - Not in BCNF:
Course(course_id, instructor, time, room)
FD: course_id → instructor
    instructor, time → room
    (instructor is not a key)

In BCNF:
Course(course_id, instructor, time)
Room(instructor, time, room)

Database Anomalies

Unnormalized databases suffer from three types of anomalies that can lead to data inconsistency:

  • Insertion Anomaly: Cannot insert certain information without other information (e.g., cannot add a department without an employee)
  • Deletion Anomaly: Deleting one piece of information causes loss of other information (e.g., deleting last employee deletes department info)
  • Update Anomaly: Changing one value requires updating multiple rows (e.g., updating department name in multiple employee records)

Keys in Relations

  • Superkey: A set of attributes that uniquely identifies a tuple
  • Candidate Key: A minimal superkey (no proper subset is a superkey)
  • Primary Key: The chosen candidate key for the relation
  • Foreign Key: An attribute that references the primary key of another relation
  • Composite Key: A key consisting of multiple attributes

Normalization Process

The normalization process involves:

  1. Identifying all functional dependencies in the relation
  2. Determining candidate keys and primary keys
  3. Checking if the relation is in 1NF, 2NF, 3NF, or BCNF
  4. Decomposing the relation if it violates normalization rules
  5. Ensuring lossless decomposition (no information loss)
  6. Preserving functional dependencies

Practical Example

Consider an unnormalized table: Student_Course(student_id, name, course_id, course_name, grade, instructor)

Functional Dependencies:
student_id → name
course_id → course_name, instructor
student_id, course_id → grade

Normalized to 3NF:
Student(student_id, name)
Course(course_id, course_name, instructor)
Enrollment(student_id, course_id, grade)
